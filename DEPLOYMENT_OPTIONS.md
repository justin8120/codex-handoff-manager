## Vite Base Path

Local builds use the default root base path:

```bash
npm run build
```

For GitHub repository pages, the deployed site is usually served from:

```text
https://<user-or-org>.github.io/<repository-name>/
```

The deploy workflow sets:

```bash
VITE_BASE_PATH=/${{ github.event.repository.name }}/
```

`vite.config.ts` reads that value with:

```ts
base: process.env.VITE_BASE_PATH ?? "/"
```

This keeps local builds working while allowing GitHub Pages repository deployments without hard-coding the repository name.

## Selected Option: GitHub Pages

### Fit

GitHub Pages is the best fit for the current project because this app is a static Vite handoff manager with no backend, database, authentication, or server runtime.

### Pros

- Built into GitHub repositories.
- No separate hosting platform is required.
- Works well for static assets generated by `npm run build`.
- Pairs naturally with the existing GitHub Actions CI setup.
- The deploy workflow can derive the repository page base path from GitHub Actions context.

### Cons

- Repository-page deployments need the correct Vite `base` path.
- Pull request preview deployments are not as rich as Vercel or Netlify.
- GitHub Pages must be enabled/configured in repository settings.

## Deploy Workflow

The deployment workflow is:

```text
.github/workflows/deploy-pages.yml
```

It runs on pushes to `main` or `master` and performs:

1. Checkout
2. Setup Node
3. Setup GitHub Pages
4. `npm ci`
5. `npm run validate`
6. `npm run build` with `VITE_BASE_PATH=/${{ github.event.repository.name }}/`
7. Upload `dist`
8. Deploy to GitHub Pages

Required permissions are configured:

- `contents: read`
- `pages: write`
- `id-token: write`

## Alternative: Vercel

Vercel remains a good option if pull request preview URLs become important or the app grows beyond a static handoff tool.

Pros:

- Excellent Vite support.
- Excellent preview deployments.
- Simple import from GitHub.

Cons:

- Requires a Vercel account and project.
- Adds an external platform dependency.
- More capability than this project currently needs.

## Alternative: Netlify

Netlify remains a good static-site option, especially if redirects or custom headers become important.

Pros:

- Strong static hosting support.
- Good preview/deploy settings.
- Simple build command and publish directory setup.

Cons:

- Requires a Netlify account and site.
- Adds an external platform dependency.
- Not necessary for the current static handoff app.
